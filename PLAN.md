# План мобильной веб-демки для сканирования «Честного ЗНАКа»

## 1. Цель демо

Сделать мобильную веб-демку, которую можно открыть с телефона по ссылке и протестировать сценарий:

1. Пользователь открывает веб-приложение на телефоне.
2. Приложение запрашивает доступ к камере.
3. Пользователь сканирует Data Matrix код маркировки на товаре.
4. Приложение распознает код и выделяет основные поля маркировки.
5. Приложение показывает карточку товара.
6. Приложение показывает дату покупки, если она найдена в демо-источнике данных.

Важно: дата покупки обычно не лежит внутри самого Data Matrix кода. Для MVP она будет браться из mock-реестра покупок. В реальном продукте этот слой нужно подключать к backend-источнику: чековые данные, OFD/ФНС, кассовая система, CRM, личный кабинет магазина или другой легальный источник транзакций.

## 2. Формат MVP

MVP делаем как мобильное веб-приложение / PWA:

- HTML, CSS, JavaScript без сборки на первом этапе;
- адаптация под экран телефона;
- доступ к камере через `getUserMedia`;
- сканирование Data Matrix через браузерный `BarcodeDetector`, где он доступен;
- fallback на JS-библиотеку для Data Matrix или ручной ввод;
- mock-данные внутри приложения;
- запуск локально через Python HTTP server;
- тест с телефона через HTTPS-хостинг.

Почему веб, а не Android APK на первом этапе:

- быстрее сделать первый рабочий прототип;
- не нужны Android Studio, Gradle, JDK и сборка APK;
- ссылку проще отправить и открыть с телефона;
- проще менять UI и mock-данные;
- можно позже завернуть в PWA или перейти к Android, если веб-сканер окажется недостаточно надежным.

## 3. Важное ограничение камеры в браузере

Камера через `getUserMedia` работает только в secure context:

- `https://...` работает;
- `http://localhost` работает на том же устройстве;
- `file://...` формально может считаться secure context, но для мобильного тестирования неудобен;
- `http://192.168.x.x` с телефона обычно не даст доступ к камере.

Поэтому для теста с телефона нужен один из вариантов:

- выложить статическую демку на GitHub Pages, Netlify, Vercel или Cloudflare Pages;
- поднять локальный HTTPS-сервер;
- использовать туннель с HTTPS, например Cloudflare Tunnel или ngrok.

Для первого MVP самый простой путь: локально разрабатываем через `python3 -m http.server`, а для проверки камеры с телефона публикуем статические файлы на HTTPS-хостинг.

## 5. Библиотеки и подход к сканированию

Первый вариант сканера:

- использовать нативный `BarcodeDetector`, если браузер поддерживает формат `data_matrix`;
- на старте приложения проверять доступные форматы через `BarcodeDetector.getSupportedFormats()`;
- если `data_matrix` недоступен, показывать fallback.

Fallback для MVP:

- ручной ввод сырой строки кода маркировки;
- загрузка фото Data Matrix с упаковки;
- позже можно подключить `@zxing/browser`, где есть Data Matrix reader.

Практичный подход:

1. Сначала сделать интерфейс, парсер, mock-данные и ручной ввод.
2. Затем подключить камеру и `BarcodeDetector`.
3. После теста на телефоне решить, нужен ли JS fallback через ZXing.

Так мы не блокируем весь MVP на качестве веб-сканирования Data Matrix, которое зависит от браузера, камеры, фокуса, освещения и размера кода.

## 7. Что показываем пользователю

Основной экран:

- верхняя панель с названием демки;
- область камеры;
- рамка наведения на Data Matrix;
- статус: «Наведи камеру на код», «Код распознан», «Проверяем»;
- кнопка ручного ввода;
- кнопка загрузки фото, если добавим этот fallback;
- сообщение, если браузер не поддерживает сканирование Data Matrix.

Экран результата:

- название товара;
- категория;
- производитель;
- GTIN;
- серийный номер;
- статус кода: например «В обороте», «Продан», «Не найден», «Подозрительный код»;
- дата и время покупки, если есть;
- магазин/точка продажи, если есть;
- подсказка, что делать при ошибке или подозрительном статусе.

Историю сканов не делаем в MVP.

## 8. Данные, которые парсим из кода

Для демо достаточно разобрать базовые GS1 Application Identifiers:

- `01` - GTIN товара, 14 цифр;
- `21` - серийный номер;
- `91` - ключ проверки;
- `92` - код проверки;
- `17` - срок годности, если присутствует;
- `10` - партия, если присутствует.

На первом этапе не пытаемся полностью валидировать криптохвост. Для демо проверяем:

- что GTIN присутствует;
- что GTIN имеет корректную длину;
- что серийный номер присутствует;
- что структура похожа на GS1 DataMatrix;
- что код найден или не найден в mock-реестре.

## 10. Mock-данные для демо

В приложении будет локальный JS-объект или JSON:

```json
{
  "gtin": "04601234567893",
  "serial": "A1B2C3D4E5",
  "productName": "Молоко пастеризованное 3.2%",
  "category": "Молочная продукция",
  "manufacturer": "ООО Демоферма",
  "status": "SOLD",
  "purchaseDate": "2026-05-21T18:42:00+10:00",
  "store": "Демо Маркет, Владивосток"
}
```

Нужны минимум 4 демо-сценария:

- успешный код, товар найден, покупка найдена;
- товар найден, но покупки нет;
- код структурно корректный, но не найден в реестре;
- код распознан, но структура некорректна.

## 11. Порядок разработки

### Этап 1. Каркас веб-приложения

- создать `index.html`;
- создать стили для мобильного интерфейса;
- создать `app.js`;
- добавить mock-данные;
- добавить README с запуском.

### Этап 2. Парсер кода маркировки

- реализовать `MarkingCodeParser`;
- выделять GTIN и серийный номер;
- учитывать разделитель GS / ASCII 29;
- поддержать несколько вариантов строки, потому что разные сканеры могут отдавать код по-разному;
- добавить тестовые строки прямо в UI или в простые JS-тесты.

### Этап 3. Ручной ввод

- добавить форму ручного ввода;
- добавить кнопки с демо-кодами;
- связать ввод с парсером и mock-реестром;
- показать результат.

### Этап 4. Камера

- запросить доступ к камере через `getUserMedia`;
- показать video preview;
- проверить наличие `BarcodeDetector`;
- проверить поддержку `data_matrix`;
- добавить цикл распознавания кадров;
- останавливать скан после успешного результата.

### Этап 5. UI результата

- сверстать карточку товара;
- добавить цветовые статусы;
- добавить блок покупки;
- добавить состояния загрузки, успеха и ошибки;
- добавить кнопку «Сканировать еще».

### Этап 6. PWA и публикация

- добавить `manifest.webmanifest`;
- добавить базовые иконки;
- опционально добавить service worker для offline shell;
- выложить на HTTPS-хостинг;
- открыть ссылку с телефона и проверить камеру.

## 12. Как запускать локально

Из корня проекта:

```bash
python3 -m http.server 5173
```

Открыть на компьютере:

```text
http://localhost:5173
```

Важно: на компьютере `localhost` подходит для проверки интерфейса и логики. Для камеры на телефоне нужен HTTPS-доступ к этой же демке.

## 13. Как тестировать с телефона

Минимальный сценарий:

1. Залить проект в GitHub.
2. Включить GitHub Pages для ветки с демкой.
3. Открыть HTTPS-ссылку на телефоне.
4. Разрешить доступ к камере.
5. Проверить сканирование реального Data Matrix.
6. Проверить ручной ввод демо-кодов.
7. Проверить сценарии «товар найден», «покупка найдена», «покупки нет», «код не найден», «плохая структура».

Если камера не открывается:

- проверить, что ссылка начинается с `https://`;
- проверить разрешения браузера;
- попробовать Chrome на Android;
- проверить, что страница не открыта внутри мессенджера, а именно в браузере;
- использовать ручной ввод как fallback.

## 14. Реальная интеграция после MVP

После демо mock-слой можно заменить на backend:

1. Веб-приложение отправляет на backend распознанный код маркировки.
2. Backend авторизуется во внешних системах и делает проверку.
3. Backend возвращает приложению нормализованную карточку:
   - товар;
   - статус;
   - признаки риска;
   - историю выбытия/покупки, если она доступна законным способом.

Не стоит хранить секреты, сертификаты и ключи доступа к интеграциям во frontend-коде. Все чувствительные интеграции должны жить на backend.

## 15. Риски и ограничения

- Поддержка `BarcodeDetector` и формата `data_matrix` зависит от браузера.
- Data Matrix на упаковках часто маленький, бликует и требует хорошего фокуса.
- Не каждый браузер одинаково хорошо работает с камерой и обработкой кадров.
- Камера требует HTTPS.
- Дата покупки не является гарантированным полем Data Matrix кода.
- Для реальной проверки статуса нужен backend и официальная интеграция.
- Для получения чековых данных могут понадобиться отдельные юридические основания, согласие пользователя или интеграция с владельцем транзакционных данных.
- Криптографическую проверку кода не делаем на клиенте в MVP.

## 16. Критерии готовности демо

Демо считаем готовым, когда:

- приложение запускается локально через `python3 -m http.server`;
- приложение открывается на телефоне по HTTPS-ссылке;
- ручной ввод работает;
- камера открывается там, где браузер поддерживает нужные API;
- минимум 4 демо-сценария показывают разные результаты;
- результат содержит товар, статус и дату покупки там, где она есть в mock-данных;
- в README описано, как запустить локально и как открыть с телефона.
